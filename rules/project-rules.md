# GBIT_GAME Project Rules

## 项目定位
这是一个持续演进中的项目，不是一次性 Demo。

实现时必须优先保证：
1. 与现有架构兼容。
2. 不破坏主循环（移动、战斗、结算、存档）。
3. 不破坏已有模式边界（玩法层 / 美术生产层 / 运行时层）。
4. 不引入难维护的临时写法。
5. 方案可扩展，能支撑后续迭代。

## Demo 周硬约束（2026-03-24 ~ 2026-03-29）
1. 技术栈锁定 Web（HTML/CSS/JS + 现有本地服务脚本）。
2. 本周内禁止 Godot 迁移或并行重写引擎。
3. 所有工作围绕“可打完一个道馆”的 Demo 闭环推进：
   - 开场引导
   - 抓宠
   - 练级 / 换宠 / 道具
   - 道馆战
   - 胜利奖励
   - 存档
4. 本周剧情与文案必须遵守 `docs/world-lore.md` 的核心约束：
   - 徽章胜利不等于神性继承
   - 三条进化道路并存，不走纯数值碾压叙事
   - Apex 与 Legendary 分层不可混淆
5. 竞赛提交必须保留来源追溯文件并持续更新：
   - `COMPLIANCE/import-manifest.json`
   - `COMPLIANCE/third-party-declaration.md`
   - `COMPLIANCE/creation-journal.md`
6. 最终提交中使用的 AI 生成产出（代码/文案/设计/素材）必须在比赛窗口 `2026-03-24 ~ 2026-03-29` 内重新生成，并保留时间与产物证据。
7. 允许复用赛前灵感与经验，但任何赛前资源（代码片段、素材、模板、提示词基线）都必须在 `COMPLIANCE/third-party-declaration.md` 显式申报来源、用途与赛期重做情况。

## 赛事创作导向（补充规则）
1. 本次评估核心是“想法与创意在 AI 助力下的有效呈现”，不是资产数量竞赛。
2. 资产量堆砌、过度包装若不能提升核心体验，应降级优先级。
3. 冲刺期优先完成“最小可玩核心体验”闭环，再考虑美术扩量与包装。
4. AI 使用应服务于拆解与表达效率：更快验证玩法、更快迭代关键体验，而非仅增加表层内容。
5. 合规记录用于保护创意与过程可追溯，不应反向挤占核心开发时间；记录以“清晰、可核验、不过度负担”为原则。

## 架构与实现约束
1. `game.js` 仅做轻量入口，不承载复杂业务。
2. 新功能优先落在 `src/game/` 对应模块。
3. 模块加载顺序必须在 `index.html` 显式维护并与依赖一致。
4. 修改存档字段时必须提供默认值与向后兼容处理。

## AI 与资源管线约束
1. 运行时图像生成使用 provider 模式（Gemini/Proxy），不依赖 ComfyUI 运行时链路；默认 provider 为 `proxy`（吉比特内网）。
2. AI 面板可用性以 `/api/ai/health -> imageProviderReady` 为准。
3. 进化立绘优先带参考图（目标/素材），但在 AI 不可用时玩法仍需可继续。
4. 融合进化与吞噬进化必须共用统一底层 skill（多图参考编辑），避免双管线分叉。
5. 玩家立绘与精灵立绘替换后，前端面板必须即时可见（含剧情立绘与队伍类卡片）。
6. 玩家立绘收藏位遵循“初始 3 + 每枚徽章 +1”，不得硬编码成固定值。
7. 功能面板展开态必须可滚动访问到底部内容，禁止因固定高度 + `overflow: hidden` 导致操作不可达。
8. NPC 立绘必须支持“回车互动触发”路径，不能仅依赖面板按钮手动触发。
9. 首遇图鉴立绘必须支持“双候选 + 玩家定稿”，并落盘到可兼容存档字段。
10. 密钥只允许存放在系统环境变量或本地 `.env.local`，严禁提交到仓库。
11. 涉及 AI provider 配置改动时，必须同步更新 `.env.example`、`docs/game-architecture.md`、`docs/system-data-flow.md`。

## 战斗系统约束
1. 每只精灵最多 4 招；学习第 5 招必须替换旧招。
2. 招式必须显式区分 `physical / special / status`。
3. 属性克制、STAB、状态异常变更需要同步 UI 可读提示。
4. 战斗中切换与道具使用能力不可回退。
5. 地区霸主（`isApex`）与传说（`isLegendary`）必须分层实现，结算奖励和文案不可混用。

## 探索系统约束
1. 关键道具不应成为单一路径硬卡点（需保留替代获取路径或绕行逻辑）。
2. 新地图门（如 O/K/P/U）必须包含：
   - 进入条件
   - 失败提示
   - 返回路径
3. 新系统（如石像、避怪丹）必须闭环到战斗/移动/存档流程。
4. 商店库存达到多分类规模时，必须使用“分类 + 分页”而非单页堆叠。
5. 新增精灵必须一次性完成四层接入：
   - `speciesData`（基础属性/进化/初始技能）
   - `speciesBattleProfiles`（双属性/成长曲线/learnset）
   - `maps.*Encounters`（出现区域/等级/权重）
   - 运行时立绘资源（`assets.generated.js` 或可回退方案）

## 编码与文本规范
1. `html/css/js/md` 必须使用 UTF-8。
2. 发现乱码按 Bug 处理：先修代码，再同步知识库。
3. 静态文案修改后需抽查关键区域：战斗按钮、功能面板、剧情卡、日志区。

## 强制知识库同步
每次完成以下任一事项后，必须立即更新知识库与 rules：
1. 完成子任务/功能。
2. 修复重要 Bug。
3. 发现新的系统交互/限制。
4. 新增或修改关键文件。
