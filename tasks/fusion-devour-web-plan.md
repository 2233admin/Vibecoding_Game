# 融合/吞噬进化独立网页：目的与计划书

## 1. 立项目的

融合进化（精灵 + 元素）与吞噬进化（精灵 + 精灵）是本项目核心玩法。当前问题是：

1. 逻辑散落在主游戏流程中，调试成本高。
2. AI 立绘生成与属性/特性/技能结果耦合，失败时不易定位。
3. 功能尚未形成“可独立运行、可回归验证、可稳定回并”的闭环。

本计划目标是先建设一个独立网页原型，先跑通完整进化链路，再无缝并回主游戏。

## 2. 方案结论（先回答你的想法）

这个方案正确，且工程上优于直接在主游戏里反复改。  
原因：可以把高风险链路（AI 图像 + 进化规则 + 结果落盘）单独验证，减少对主循环的干扰。

## 3. 范围定义

## In Scope（本期必须完成）

1. 独立网页页面（Fusion Lab）。
2. 两种模式：
   - 融合进化：`精灵 + 元素`
   - 吞噬进化：`精灵 + 精灵`
3. 统一输出：
   - 新属性（type）
   - 新特性（trait）
   - 新技能集（skills，遵守最多 4 招）
   - 新立绘（AI 生成）
4. 可回放的任务日志（请求参数、任务状态、失败原因）。
5. 结果可写入与主游戏兼容的数据结构（便于回并）。

## Out of Scope（本期不做）

1. 地图、道馆、主线任务联动。
2. 全量 UI 美术重做。
3. 复杂多人/联网对战。

## 4. MVP 功能清单

1. 基础输入区：
   - 主体精灵选择
   - 元素或素材精灵选择
   - 可编辑提示词（prompt）
2. 规则计算区：
   - 实时预览属性/特性/技能变化
   - 显示技能替换策略（第 5 招触发遗忘选择）
3. AI 生成区：
   - 调用统一图像生成接口
   - 展示生成中、成功、失败状态
4. 结果区：
   - 新精灵卡片（名称、属性、特性、技能、立绘）
   - 一键“写入本地结果集”
5. 历史区：
   - 最近 N 条进化记录
   - 支持复用参数再次生成

## 5. 技术方案（独立运行 + 可回并）

## 5.1 独立入口

新增页面入口（建议）：

1. `fusion-lab.html`（独立网页）
2. `src/fusion-lab/`（独立模块目录）

## 5.2 模块拆分

建议拆分为以下模块：

1. `src/fusion-lab/state.js`：页面状态机（idle/running/success/failed）
2. `src/fusion-lab/rules-engine.js`：属性/特性/技能计算（纯函数）
3. `src/fusion-lab/image-service.js`：AI 图像任务封装（提交+轮询）
4. `src/fusion-lab/storage.js`：结果存档（local json / runtime manifest）
5. `src/fusion-lab/ui.js`：渲染与事件绑定
6. `src/fusion-lab/main.js`：启动入口

## 5.3 接口与数据契约

统一进化结果对象（建议）：

```json
{
  "mode": "fusion | devour",
  "baseSpeciesId": "sproutle",
  "material": { "type": "element | species", "id": "fire" },
  "result": {
    "speciesId": "sproutle_fire",
    "name": "焰芽灵",
    "types": ["grass", "fire"],
    "trait": "火共鸣",
    "skills": ["leaf_cut", "ember", "growth_boost", "guard_seed"],
    "portraitKey": "monsters:sproutle_fire_v1"
  },
  "artTask": {
    "taskId": "task_xxx",
    "status": "queued | running | completed | failed"
  },
  "createdAt": "2026-03-20T00:00:00.000Z"
}
```

## 5.4 与主游戏的兼容点

1. 规则引擎纯函数化，后续由 `src/game/evolution-pipeline.js` 直接复用。
2. 结果字段兼容现有 `mutation` 结构（含 `trait` 和 `portraitKey`）。
3. 图像任务继续复用现有 `/api/ai/generate-monster` 与 `/api/ai/tasks/:id`。

## 6. 实施里程碑

## Milestone 1：骨架搭建（1 天）

1. 建立 `fusion-lab.html` 和 `src/fusion-lab/*` 目录。
2. 完成页面状态管理与基础 UI。
3. 可本地模拟一条假进化结果。

验收：页面可独立打开并完成一次“假数据进化”。

## Milestone 2：规则跑通（1-2 天）

1. 融合/吞噬两套规则接入统一规则引擎。
2. 产出属性、特性、技能结果。
3. 接入 4 招上限与技能替换策略。

验收：输入变化后，结果稳定且可重复。

## Milestone 3：AI 立绘跑通（1-2 天）

1. 对接 AI 生成任务提交+轮询。
2. 任务日志可视化。
3. 失败重试与降级占位图。

验收：至少 3 组样例可成功生成并展示立绘。

## Milestone 4：回并主游戏（1 天）

1. 规则引擎回并到 `src/game/evolution-pipeline.js`。
2. UI 层在主游戏里只做调用，不重复写规则。
3. 增加回归验收脚本用例。

验收：主游戏中融合/吞噬进化稳定可用，不破坏主循环。

## 7. 验收标准（Definition of Done）

1. 独立网页可独立运行，且两种进化模式都可完成闭环。
2. 每次进化都会产生：新属性、新特性、新技能、新立绘。
3. 失败有可见日志与可恢复策略（重试/降级）。
4. 输出数据可直接被主游戏读取。
5. 至少有一轮自动化交互验收通过。

## 8. 风险与对策

1. AI 任务不稳定：
   - 对策：任务轮询超时、失败重试、占位图降级。
2. 规则膨胀导致难维护：
   - 对策：规则引擎纯函数 + 单元测试样例。
3. 回并冲突：
   - 对策：先定义统一数据契约，再做模块替换，不在主游戏复制逻辑。

## 9. 本期输出物

1. 独立网页原型（Fusion Lab）。
2. 规则引擎初版。
3. AI 图像任务封装层。
4. 回并适配说明。

## 10. 下一步执行建议

按里程碑从 M1 开始落地，不跨阶段并行改核心逻辑。  
先把“可独立跑通”作为第一优先级，再把 UI 美术优化和玩法拓展放到第二阶段。
